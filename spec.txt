1. VISIÓN GENERAL Y OBJETIVOS DEL SISTEMA
1.1 Definición del problema
El market making algorítmico es el proceso de colocar simultáneamente órdenes de compra (bid) y venta (ask) en un libro de órdenes con el objetivo de capturar el spread bid-ask como ganancia, mientras se gestiona el riesgo de inventario que surge cuando el mercado se mueve de forma adversa. El desafío central es que el market maker enfrenta un trade-off fundamental:
Spreads más amplios → mayor ganancia por trade, menor probabilidad de ejecución.
Spreads más estrechos → mayor probabilidad de ejecución, menor margen, mayor riesgo de adverse selection.
El enfoque clásico (Avellaneda-Stoikov, 2008) proporciona una solución analítica óptima bajo supuestos simplificadores. Este sistema usa Avellaneda-Stoikov como prior teórico y entrena un agente DRL para aprender las desviaciones del óptimo en condiciones reales de mercado donde los supuestos se violan continuamente.

2.2 Espacio de acción
El espacio de acción es continuo 3D, lo que permite al agente ajustar simultáneamente tres dimensiones del quote:
a = (δ_bid, δ_ask, q)  ∈  [0, 5σ] × [0, 5σ] × [0, q_max]
donde σ es la desviación estándar de precios en la última ventana de 60 segundos, y q_max es el tamaño máximo de orden configurado. El agente puede colocar quotes asimétricas (δ_bid ≠ δ_ask), lo que le permite gestionar inventario sin cancelar órdenes.
2.2.1 Discretización opcional (modo benchmark)
Para comparación con baselines, se implementa un espacio discreto de 125 acciones = 5 niveles bid × 5 niveles ask × 5 tamaños. Usar gym.spaces.Discrete(125) con un mapping idx → (δ_bid, δ_ask, q).
2.3 Función de recompensa — reward shaping crítico
La función de recompensa es el componente más delicado del sistema. Una especificación incorrecta produce políticas degeneradas (no cotizar nunca, o cotizar spreads infinitos). La función combina tres términos:
R(t) = PnL_realized(t)  −  λ·inventory(t)²  −  κ·cost_taker(t)  +  ρ·fill_reward(t)
Término
Valor típico
Descripción
PnL_realized
variable
(ask_fill - avg_bid_cost) × qty cuando se completa un ciclo bid→ask. Incluye maker rebate de Binance (0.02% del nocional).
λ·inventory²
λ = 0.001
Penalización cuadrática. Asimétrica: λ_long = 0.001, λ_short = 0.0015 (posiciones cortas más penalizadas en cripto por funding). Se eleva a λ·inventory⁴ cuando |inventory| > 0.8·max_inventory.
κ·cost_taker
κ = 0.0007
Penalización cuando el agente genera órdenes market (taker). Fee de Binance: 0.07% spot, 0.04% futuros. Incentiva órdenes limit (maker).
ρ·fill_reward
ρ = 0.0001
Reward shaping pequeño por cada fill exitoso. Evita la política degenerada de no cotizar. Debe ser menor al PnL esperado para no crear incentivos incorrectos.
CRÍTICO — Normalización de la recompensa: dividir R(t) por la volatilidad realizada del periodo (σ_30s) para obtener un ratio tipo Sharpe por step. Esto estabiliza el entrenamiento cuando la volatilidad del mercado cambia en órdenes de magnitud.
2.3.1 Pseudocódigo de la función de recompensa
def compute_reward(
    inventory: float,          # posición neta normalizada [-1, 1]
    realized_pnl: float,       # PnL realizado en el step
    taker_cost: float,         # costo de órdenes taker (siempre >= 0)
    fill_occurred: bool,       # si hubo ejecución este step
    sigma_30s: float,          # volatilidad realizada 30s
    lambda_base: float = 0.001,
    kappa: float = 0.0007,
    rho: float = 0.0001,
) -> float:
    # Penalización cuadrática asimétrica
    if inventory > 0:
        inv_penalty = lambda_base * inventory ** 2
    else:
        inv_penalty = 1.5 * lambda_base * inventory ** 2
    # Escalado cuártico para inventario extremo  # CRÍTICO
    if abs(inventory) > 0.8:
        inv_penalty *= (abs(inventory) / 0.8) ** 2
    fill_bonus = rho if fill_occurred else 0.0
    raw_reward = realized_pnl - inv_penalty - kappa * taker_cost + fill_bonus
    # Normalización por volatilidad  # CRÍTICO para estabilidad
    return raw_reward / (sigma_30s + 1e-8)

3.2 Arquitectura de red neuronal
3.2.1 Spread agent — S4 (Structured State Space Sequence Model)
Usar S4 en lugar de LSTM por su capacidad de capturar dependencias de largo alcance en O(N log N) versus O(N²) de la atención. Crítico para order flow donde eventos de hace 500ms pueden ser altamente predictivos.
class SpreadAgentNetwork(nn.Module):
    def __init__(self, obs_dim: int = 47, action_dim: int = 3,
                 d_model: int = 256, n_layers: int = 4, seq_len: int = 64):
        # Input projection
        self.input_proj = nn.Linear(obs_dim, d_model)
        # S4 layers para dependencias temporales
        self.s4_layers = nn.ModuleList([S4Layer(d_model) for _ in range(n_layers)])
        # Actor head (Gaussian policy)
        self.actor_mean = nn.Linear(d_model, action_dim)
        self.actor_log_std = nn.Parameter(torch.zeros(action_dim))
        # Critic head (twin Q-networks para SAC)
        self.critic_q1 = nn.Linear(d_model + action_dim, 1)
        self.critic_q2 = nn.Linear(d_model + action_dim, 1)
    def forward(self, obs_seq: torch.Tensor) -> tuple[Tensor, Tensor]:
        # obs_seq shape: (batch, seq_len, obs_dim)
        x = self.input_proj(obs_seq)          # (B, T, d_model)
        for layer in self.s4_layers:
            x = layer(x) + x                  # residual connection
        x_last = x[:, -1, :]                  # tomar último estado
        mean = torch.tanh(self.actor_mean(x_last))
        std  = self.actor_log_std.exp().clamp(1e-3, 1.0)
        return mean, std
3.2.2 Meta-controller — MAML (Model-Agnostic Meta-Learning)
El meta-controller se entrena con MAML para adaptarse rápidamente (en 5-10 gradient steps) a un nuevo régimen de mercado usando solo los últimos 500 trades como soporte. Esto elimina la necesidad de reentrenar desde cero cuando el mercado cambia de régimen.
# Proceso MAML (inner loop)
def maml_inner_loop(
    meta_params: dict,
    support_batch: Batch,   # últimos 500 trades del nuevo régimen
    inner_lr: float = 0.01,
    n_inner_steps: int = 5,
) -> dict:
    adapted_params = meta_params.copy()
    for _ in range(n_inner_steps):
        loss = compute_regime_loss(adapted_params, support_batch)
        grads = torch.autograd.grad(loss, adapted_params.values())
        adapted_params = {k: v - inner_lr * g
                          for (k, v), g in zip(adapted_params.items(), grads)}
    return adapted_params

4. CAPA 5 — WORLD MODEL Y SIMULACIÓN (MBRL)
4.1 Dreamer-v3 para market making
El problema fundamental del DRL en trading es la eficiencia de datos. El agente no puede explorar libremente — cada exploración tiene costo real. Dreamer-v3 (Hafner et al., 2023) resuelve esto entrenando un world model que predice la dinámica del mercado, permitiendo rollouts completamente sintéticos ('en imaginación').
Ganancia de eficiencia: en experimentos de mercados equitativos, Dreamer-v3 alcanza el mismo nivel de política que PPO puro con ~10x menos datos reales, y ~50x menos steps en entornos con dinámica compleja.
4.1.1 Componentes del world model
El world model de Dreamer-v3 tiene 4 redes neuronales entrenadas conjuntamente:
Representation model: q(z_t | z_{t-1}, a_{t-1}, o_t) — codifica observaciones reales en estados latentes z_t usando RSSM (Recurrent State Space Model).
Transition model: p(z_t | z_{t-1}, a_{t-1}) — predice estados latentes futuros dado un estado y acción, SIN ver la observación real. Crítico para los rollouts sintéticos.
Reward model: p(r_t | z_t) — predice el reward desde el estado latente.
Decoder (opcional en inferencia): p(o_t | z_t) — reconstruye la observación para regularización del entrenamiento del world model.

# Fase 1: Recolección de datos reales (mínimo viable)

4.2 Neural LOB Simulator
Complementando a Dreamer-v3, se entrena un simulador de libro de órdenes dedicado con arquitectura Mamba (Gu & Dao, 2023) — los State Space Models son superiores a Transformers para secuencias largas de ticks porque su complejidad es O(N) versus O(N²).
El simulador aprende la distribución condicional del próximo estado del LOB dado el estado actual y la acción del market maker, incluyendo el impacto de precio (slippage) modelado según Almgren-Chriss:
ΔP_impact = η × sign(Q) × |Q|^δ  +  ε_t
donde 
η
 es el coeficiente de impacto permanente (calibrado con datos históricos), 
δ ≈ 0.6
 es el exponente de impacto de mercado (Bacry et al., 2015), y 
ε_t
 es ruido de mercado.

5.3 Cálculo de VPIN
VPIN (Volume-weighted Probability of Informed Trading, Easley et al., 2012) es el feature de microestructura más predictivo de crashes de liquidez. El cálculo correcto es:
VPIN = (1/n) × Σ_{k=1}^{n} |V_buy_k − V_sell_k| / V_bucket
donde cada bucket k acumula V_bucket en volumen de transacciones, y V_buy/V_sell se estima clasificando cada trade con el algoritmo de Lee-Ready (comparación con mid-price).
fn compute_vpin(
    buckets: &VecDeque<VpinBucket>,
    window: usize,  // número de buckets (típicamente 50)
) -> f64 {
    let n = buckets.len().min(window);
    let sum: f64 = buckets.iter().rev().take(n)
        .map(|b| (b.buy_vol - b.sell_vol).abs() / b.total_vol)
        .sum();
    sum / n as f64
}
6. BACKTESTING REALISTA Y GESTIÓN DE RIESGOS
6.1 Motor de backtesting con microestructura real
El backtesting naive (fill al precio teórico) sobrestima el PnL real por factores de 2x a 5x. El simulador debe modelar explícitamente:
Factor
Implementación requerida
Latencia de red
Samplear latencia de distribución log-normal: μ=2ms, σ=0.5ms para co-location. Aplicar como delay entre decisión y envío de orden.
Latencia de matching engine
Adicional 0.1ms-1ms en el matching engine de Binance. Modelar con distribución exponencial.
Cola del order book
Una orden limit en nivel 1 NO se ejecuta instantáneamente. Se asume posición al final de la cola. Fill cuando el volumen acumulado previo es ejecutado.
Adverse selection
Si una orden es tomada en < 50ms de su colocación: 40% probabilidad de ser adverse (el precio se moverá en contra). Aplicar penalización adicional de 0.5 bps.
Impacto de precio
Para órdenes > 0.1% del volumen medio del nivel 1, aplicar modelo Almgren-Chriss. Slippage = η × (Q/V_avg)^δ × σ.
Cancelaciones parciales
Modelar que el 5-15% de las órdenes limit propias son 'sniped' (canceladas por otro participante que las detecta) antes de llegar al matching engine.
6.2 Sistema de circuit breakers
El circuit breaker es el sistema de seguridad crítico. Debe operar independientemente del agente DRL — si el agente falla o produce acciones degeneradas, el circuit breaker detiene el sistema y liquida posiciones.
class CircuitBreaker:
    def __init__(
        self,
        max_drawdown_pct: float = 0.05,   # 5% del capital
        max_inventory_notional: float = 10_000,  # USD
        vol_multiplier: float = 3.0,      # 3x vol histórica = flash crash
        max_position_age_s: float = 300,  # 5 min max holding
        daily_loss_limit: float = 0.02,   # 2% capital/día
    ):
        ...
    def check(self, state: MarketState, portfolio: Portfolio) -> Action:
        # Trigger 1: Drawdown desde peak
        if portfolio.drawdown_from_peak() > self.max_drawdown_pct:
            return Action.EMERGENCY_LIQUIDATE
        # Trigger 2: Volatilidad extrema (flash crash)  # CRÍTICO
        current_vol = state.realized_vol_1min
        if current_vol > self.vol_multiplier * state.historical_vol_30d:
            return Action.CANCEL_ALL_PAUSE_30S
        # Trigger 3: Inventario stale
        if portfolio.max_position_age() > self.max_position_age_s:
            return Action.FORCE_FLATTEN
        # Trigger 4: Pérdida diaria acumulada
        if portfolio.daily_pnl() < -self.daily_loss_limit * portfolio.capital:
            return Action.SHUTDOWN_TODAY
        return Action.CONTINUE
